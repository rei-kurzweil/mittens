# MMS Language Server & VSCode Extension

## The four independent pieces

Building proper editor support for `.mms` decomposes into four things that can
be built and shipped independently:

| Piece | Format / language | Needed for |
|---|---|---|
| TextMate grammar | JSON (regexes) | Syntax highlighting in VSCode + GitHub |
| VSCode extension manifest | `package.json` + TypeScript | Wiring everything together |
| Language server | Rust (`tower-lsp`) | Hover, completions, diagnostics, go-to-def |
| API manifest | JSON/TOML, generated by `build.rs` | Telling the server about components |

They are independent: you can ship syntax highlighting with zero language server,
and add the server later. GitHub only uses the TextMate grammar.

---

## 1. TextMate grammar

A single JSON file. VSCode consumes it for syntax highlighting; GitHub Linguist
also accepts it (referenced from `.gitattributes` or a Linguist PR).

Structure is a set of named regex rules (`patterns`) with a `scopeName` like
`source.mms`. Scopes map to theme colours.

### What needs to be captured for MMS

**Keywords** (from `token.rs`):
```
let  if  else  return  true  false  null  fn  for  in  break  continue
```

**Component expressions** — the grammar's main syntactic shape:
- `TypeName.constructor(args) { body }` — uppercase-leading ident followed by
  `.` is the most distinctive pattern. Scope it as `entity.name.type`.
- `TypeName { body }` — same ident without a dot.
- Body calls `method(args)` — lowercase ident followed by `(` inside a brace
  block. Scope as `entity.name.function`.
- Constructor method `TypeName.method(` — the `.method` part is
  `entity.name.function`.

**Shortforms** — single-letter and all-caps identifiers like `T`, `R`, `C`,
`TXT`, `AVC`. These are just identifiers; the grammar does not need to enumerate
them. The `entity.name.type` rule (uppercase-leading ident) catches them all.

**Literals:** strings `"..."`, numbers (including leading `-`), arrays `[...]`,
booleans `true`/`false`, `null`.

**Comments:** `//` line comments.

### Why not generate the grammar from Rust source?

The grammar is about **syntax shape**, not about **what names exist**. The
patterns that need to be written are:

- One regex for `//` comments.
- One regex for string literals.
- One regex for numbers.
- One regex for uppercase-leading identifiers (component type names) — this
  catches all current and future component names without enumerating them.
- One regex for lowercase `method(` calls inside bodies.
- A keyword alternation for the ~12 keywords.

None of these require reading the Rust source. The shortform table
(`COMPONENT_SHORTFORMS` in `token.rs`) is an all-caps identifier subset of the
same uppercase-leading rule — it is fully covered by the general pattern.

The only thing that *could* come from generated source is an explicit keyword
list for component type names (to give them a distinct colour from arbitrary
identifiers). But that's a quality-of-life nicety, not required for correctness,
and maintaining a generated keyword list inside a grammar file adds build
complexity for negligible gain.

**Conclusion: write the grammar directly.** It is approximately 80–120 lines of
JSON. Claude Code can produce a correct first draft of it in one pass faster than
building the machinery to generate it.

## 2. VSCode extension manifest (`package.json`)

A VSCode extension is a directory with a `package.json` that declares:

```json
{
  "contributes": {
    "languages": [{
      "id": "mms",
      "extensions": [".mms"],
      "configuration": "./language-configuration.json"
    }],
    "grammars": [{
      "language": "mms",
      "scopeName": "source.mms",
      "path": "./syntaxes/mms.tmLanguage.json"
    }]
  }
}
```

`language-configuration.json` declares comment characters (`//`), bracket
pairs, and auto-closing pairs — a dozen lines.

The extension also launches the language server if one exists. That is a
TypeScript file (~50 lines) using `vscode-languageclient`:

```ts
import { LanguageClient, TransportKind } from 'vscode-languageclient/node';

const client = new LanguageClient('mms', 'MMS Language Server', {
    command: context.asAbsolutePath('bin/mms-language-server'),
    transport: TransportKind.stdio,
}, { documentSelector: [{ scheme: 'file', language: 'mms' }] });

client.start();
```

The TypeScript side knows nothing about MMS — it just spawns the binary and
proxies JSON-RPC.

## 3. Language server (Rust + `tower-lsp`)

LSP is a **language-agnostic JSON-RPC protocol** over stdin/stdout. The server
is a subprocess; any language works. Writing it in Rust makes sense because:

- The MMS tokenizer and parser already exist in the engine crate.
- The server can depend on `cat-engine` as a library and reuse them directly.
- `tower-lsp` provides the async scaffolding; you implement trait methods.

Minimal server skeleton:

```rust
#[tower_lsp::async_trait]
impl LanguageServer for MmsServer {
    async fn initialize(&self, _: InitializeParams) -> Result<InitializeResult> { ... }
    async fn did_open(&self, params: DidOpenTextDocumentParams) { /* parse, store */ }
    async fn did_change(&self, params: DidChangeTextDocumentParams) { /* re-parse */ }
    async fn completion(&self, params: CompletionParams) -> Result<Option<CompletionResponse>> { ... }
    async fn hover(&self, params: HoverParams) -> Result<Option<Hover>> { ... }
    async fn goto_definition(&self, params: GotoDefinitionParams) -> Result<Option<GotoDefinitionResponse>> { ... }
}
```

Features can be shipped incrementally:

1. **Diagnostics only** (parse errors, unknown component names) — useful immediately.
2. **Hover** — show component docs on `T`, `AVC`, etc.
3. **Completions** — body call names, constructor methods, known component types.
4. **Go-to-definition** — for `let` bindings and (once the module system exists) imported names.

## 4. API manifest — how the server learns about components

The language server needs to know which component type names exist, what
constructors they accept, and what body-call methods they support. There are
three strategies:

### A. Generated by `build.rs` (recommended)

A `build.rs` script parses `component_registry.rs` (or reads a hand-maintained
source-of-truth file) and emits a JSON manifest:

```json
{
  "Transform":      { "constructors": ["position", "scale", "rotation_euler"], "body_calls": ["position", "scale"] },
  "AvatarControl":  { "body_calls": ["head_bone", "camera_bone", "left_hand_bone", ...] },
  "TextBackground": { "body_calls": ["padding", "padding_top", "padding_right", ...] }
}
```

The manifest is embedded in the server binary (`include_str!`) or shipped as a
sidecar file. Completions and hover docs are driven entirely from it. Because it
is generated from the same registry that executes at runtime, it can never drift.

### B. Server links against the engine crate directly

The server binary depends on `cat-engine` and calls into `component_registry`
at runtime to enumerate known type names and methods. Tightest coupling but
zero manifest maintenance.

### C. Hand-maintained schema

Fine during prototyping, painful at scale.

## Module system / cross-file awareness

Once MMS has imports, the server needs a **workspace index**:

- On workspace open: parse all `.mms` files, build a symbol table per file.
- On file save: re-parse the changed file, update the index.
- `textDocument/definition` resolves the symbol through the index.
- `textDocument/completion` merges: local scope + imported symbols + built-in manifest.

The parser already handles the heavy lifting; the server just drives it as a
library and routes results into LSP response types.

---

## Recommended build order

1. **TextMate grammar** — write directly (no generation needed), ~100 lines JSON.
   Immediate payoff: proper highlighting in VSCode and on GitHub.
2. **`language-configuration.json`** — 15 minutes, enables comment toggling and
   bracket matching.
3. **`build.rs` API manifest** — generates the component/method table from the
   registry. Needed before completions are useful.
4. **Language server skeleton** — diagnostics + completions from manifest.
5. **Hover docs, go-to-definition** — add as the module system matures.
