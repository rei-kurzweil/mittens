# AVC capsule inference uses the unresolved GLTF placeholder triangle

## Status

Implemented in the engine; pending runtime validation against Bisket's displayed
bounds and capsule diagnostics.

The implementation registers imported CPU meshes before simulation-time geometry
consumers run, treats `MeshComponent` as authoritative during bounds measurement,
and prevents AVC from creating a capsule while any imported primitive is pending.

## Symptom

The collision capsule generated by `AvatarControl` can be much shorter than the
avatar's aggregate render bounds. In `secondary-motion-desktop`, Bisket's visible
bounds are vertically correct, but the inferred capsule covers only the dimensions
of the built-in 2D triangle.

This also makes the apparent grounding gap misleading: the capsule's bottom
hemisphere is inferred from the wrong bounds before the imported GLTF meshes are
available to the bounds calculation.

## Evidence

The AVC inference diagnostics reported:

```text
[AVC][capsule][inferred] avc=ComponentId(977v1) model_root=ComponentId(978v1) source=aggregate_render_bounds bounds_min=[-0.5, -0.28867513, -4.371139e-8] bounds_max=[0.5, 0.57735026, 4.371139e-8] center_y=0.144338 radius=0.280000 half_segment=0.153013 local_bottom=-0.288675 local_top=0.577350 movement_target=Some(ComponentId(976v1))
```

Those bounds are the built-in `TRIANGLE_2D`, including its effectively zero Z
depth. They are not Bisket's aggregate geometry.

## Root cause

GLTF primitives are initially represented by:

- a `RenderableComponent` whose mesh is `CpuMeshHandle(0)`
- a descendant `MeshComponent` containing the actual imported-mesh key

Handle zero is the valid built-in `TRIANGLE_2D`. It is used as a structural
placeholder because `Renderable` currently requires a concrete mesh handle.
`RenderableSystem::flush_pending` later treats `MeshComponent` as authoritative,
waits for the imported mesh, and replaces both `mesh` and `base_mesh`.

AVC runs before that replacement. `BoundsSystem::calculate_subtree_local_bounds`
currently reads `Renderable.mesh` directly and therefore mistakes the placeholder
triangle for resolved geometry.

The frame ordering exposes the problem:

1. `GLTFSystem::tick_with_queue` creates the GLTF component hierarchy and emits
   `GltfInitialized`.
2. AVC requests aggregate subtree bounds during the same simulation tick.
3. `SystemWorld::prepare_render` registers imported CPU meshes and flushes pending
   renderables only after the simulation tick.

The later bounds visualization is correct because it observes resolved imported
meshes. The capsule is not recomputed after its one-time creation.

## Desired behavior

AVC must infer its capsule from the complete aggregate bounds of the actual imported
CPU meshes. It must never treat an unresolved placeholder as model geometry or
commit a capsule from a partially resolved imported subtree.

Bounds measurement should depend on CPU asset readiness, not GPU upload or visual
instance creation.

## Implementation direction

### Make imported CPU meshes available before geometry consumers run

After GLTF spawning, register decoded CPU meshes in `RenderAssets` before AVC and
other simulation-time bounds consumers run. Reuse the existing
`GLTFSystem::flush_mesh_imports_only` path; texture upload and GPU mesh upload remain
in `prepare_render`.

This separates the two readiness stages:

- CPU mesh ready: bounds, collision inference, inspection, and headless systems may run
- GPU mesh ready: the renderable may be inserted into `VisualWorld`

### Make `MeshComponent` authoritative during bounds resolution

When measuring a `RenderableComponent`:

- if it has a `MeshComponent`, resolve that key through `RenderAssets`
- if the key resolves, measure the imported CPU mesh
- if the key does not resolve, mark the aggregate measurement as pending
- only use `Renderable.mesh` directly when there is no `MeshComponent` override

Do not fall back from an unresolved `MeshComponent` to the placeholder renderable
handle.

### Represent readiness explicitly

Change aggregate bounds measurement to distinguish at least:

- `Measured(Aabb)`: every relevant renderable was resolved and contributed
- `Pending`: at least one import-backed renderable is unresolved
- `Unmeasurable`: the subtree has no measurable render geometry

AVC should retry on later ticks while bounds are `Pending`. It may use its existing
avatar-height fallback only for a genuinely `Unmeasurable` spawned model, not for a
partially loaded one.

No new `BoundsCalculated` event is required. Bounds calculation is synchronous; the
missing readiness belongs to imported mesh registration. Polling the explicit
measurement state keeps initialization robust across load timing and avoids one-shot
event ordering problems.

## Acceptance criteria

- Bisket's generated AVC capsule uses the full aggregate imported-render bounds, not
  the triangle dimensions.
- The capsule's inferred local bottom and top correspond to the measured aggregate
  bounds according to the existing upright-capsule inference policy.
- An unresolved or partially resolved imported subtree does not create a capsule.
- Built-in renderables without `MeshComponent` overrides continue to measure from
  their authored mesh handles.
- CPU bounds measurement works before GPU upload and in headless execution.
- Existing capsule radius, upright Y-capsule inference, collision response,
  grounding, and movement-target behavior remain unchanged.
- Once created from complete bounds, the collider visualization matches the capsule
  used by collision simulation.

## Tests

- Create an import-backed renderable whose placeholder is `TRIANGLE_2D` and whose
  registered mesh is tall; verify bounds use the imported mesh.
- Leave its mesh key unresolved and verify measurement returns `Pending`, with no
  triangle bounds and no AVC capsule.
- Register the mesh later, tick again, and verify AVC creates exactly one capsule
  from the resolved bounds.
- Use a subtree containing multiple imported primitives and verify no result is
  committed until all relevant keys resolve.
- Mix resolved imported meshes and ordinary built-in renderables and verify the full
  transformed union.
- Verify `Unmeasurable` remains distinct from `Pending` so the avatar-height fallback
  cannot mask load latency.
- Re-run `secondary-motion-desktop` with collider and bounds visualization enabled
  and compare the logged aggregate bounds, capsule extents, and visible model bounds.

## Non-goals

- Retuning Bisket's authored scale, capsule radius, or avatar height
- Changing the upright capsule inference formula
- Changing collision grounding or resolving unrelated camera collision setup in
  `bisket-vr-demo`
- Replacing `Renderable.mesh` with an optional or enum-based resolution state as part
  of this focused fix; that is a reasonable follow-up refactor
