# MMS Panel Click Bridge

## Goal

Move editor panels to MMS-authored structure while keeping click binding practical
during the first integration pass.

The panel factory should be free to render the full UI for a world panel or inspector panel from an item list. Clicks on those rendered items still need to call Rust-side editor operations such as:

- selecting a scene node
- focusing or replacing gizmos
- opening an inspector target
- triggering editor-only commands

## Current Direction

`InspectorSystem` now treats the MMS asset as the source of panel structure.

- Rust computes a panel item list.
- Rust evaluates the MMS factory into the live world.
- Rust swaps the previous panel subtree for the new subtree.

This is intentionally a full-panel rerender path. It is appropriate for topology or selection changes, and it keeps the first integration small.

## Constraint

The click action is not just UI-local state.

For a world-panel row, the click target maps to a scene component or editor action.
That means the handler needs a way to ask the editor to do something, but we do not
want to expose a large direct host API to MMS.

## V1 Bridge

Use Rust-side post-render binding.

1. MMS owns structure, visual state, and deterministic row naming.
2. Rust owns click binding and editor execution.

The bridge is a stable item identity carried through render.

### Render Contract

Each rendered item should have:

- a display label
- a predictable row node name queryable from the rendered content root

V1 row identity:

- render-order names such as `item_0`, `item_1`, `item_2`

Later row identity:

- a stable key generated by Rust
- a row node name derived from that key
- a target ref for the represented component or editor target
- optional visual flags such as selected, disabled, or depth

The exact later data shape can change. The important part for v1 is that Rust can
reliably query rows from the rendered subtree and attach handlers without needing
structured MMS item values yet.

## V1 Click Flow

Recommended flow for item clicks:

1. Rust builds a panel view-model list.
2. Rust evaluates the MMS factory with the current string items.
3. MMS renders rows with predictable names under the content root.
4. Rust queries those rows from the rendered subtree.
5. Rust registers the click handlers against the queried row components.

## Why This Split Is Useful

It keeps rerender semantics clean.

- world changes cause rerender
- selection changes cause rerender
- scrolling does not cause rerender
- clicking does not require rerender unless it changes editor state

This keeps the first implementation small and avoids blocking panel bring-up on
records/maps interop.

## Suggested Near-Term Item Shape

The current panel factories take `items: [String]`. That is good enough for structural bring-up, but not for robust interaction.

The next useful step is to let Rust pass item metadata, not just labels.

Minimum useful fields:

- `key`
- `label`
- `target_ref`
- `selected`
- `depth` for world tree indentation

Until structured item values exist, the current experiment should use queryable
row names plus Rust-side handler binding.

## Non-Goals For The First Pass

- Do not make scroll position or pointer hover drive rerenders.
- Do not expose the full editor API directly to MMS handlers yet.
- Do not require raw `ComponentId` values when a canonical target-ref string works.

## Later Bridge

Once structured item values exist, the next step is to spec the first editor intents
and let MMS item handlers emit them directly.

The first one should be:

- `EDITOR_SELECT(target_ref)`

That would give full MMS-authored structure and locally-authored click behavior,
while keeping the editor execution path on the Rust side.
