# MMS Panel Click Bridge

## Goal

Move editor panels to MMS-authored structure without forcing MMS event handlers to own editor-side behavior.

The panel factory should be free to render the full UI for a world panel or inspector panel from an item list. Clicks on those rendered items still need to call Rust-side editor operations such as:

- selecting a scene node
- focusing or replacing gizmos
- opening an inspector target
- triggering editor-only commands

## Current Direction

`InspectorSystem` now treats the MMS asset as the source of panel structure.

- Rust computes a panel item list.
- Rust evaluates the MMS factory into the live world.
- Rust swaps the previous panel subtree for the new subtree.

This is intentionally a full-panel rerender path. It is appropriate for topology or selection changes, and it keeps the first integration small.

## Constraint

The click action is not just UI-local state.

For a world-panel row, the click target usually maps to an existing `ComponentId` or an editor command. That means the final action needs host access, not just a script-local closure.

If MMS owns the entire click body directly, the script ends up needing more editor API surface than we want to expose immediately.

## Proposed Bridge

Use a split ownership model.

1. MMS owns structure and visual state.
2. Rust owns semantic click actions.

The bridge is a stable item identity carried through render.

### Render Contract

Each rendered item should have:

- a stable key generated by Rust
- a display label
- optional visual flags such as selected, disabled, or depth
- a predictable row node name derived from the key

Example direction:

- Rust item key: `node_123`
- MMS row root name: `item_node_123`

The exact data shape can change later. The important part is that Rust can recover the rendered row node by a deterministic selector after the panel subtree is spawned.

## Click Flow

Recommended flow for item clicks:

1. Rust builds a panel view-model list.
2. Rust evaluates the MMS factory with those items.
3. Rust queries the rendered subtree for each row node using the stable item key.
4. Rust registers a scoped `Click` handler for that row node in `RxWorld`.
5. The handler closes over the editor action payload already known in Rust.

That payload can be:

- `ComponentId` for selection
- enum-like action id for buttons or inspector commands
- gizmo command parameters

On click, the handler performs host-side behavior directly.

Examples:

- world row click: `select_editor_target(world, emit, editor_root, target_id, false)`
- inspector button click: emit or call the relevant editor command
- gizmo mode row click: mutate editor/gizmo state and let selection-driven rerender update visuals later

## Why This Split Is Useful

It keeps rerender semantics clean.

- world changes cause rerender
- selection changes cause rerender
- scrolling does not cause rerender
- clicking does not require rerender unless it changes editor state

This also avoids making MMS responsible for long-lived editor state coordination too early.

## Suggested Near-Term Item Shape

The current panel factories take `items: [String]`. That is good enough for structural bring-up, but not for robust interaction.

The next useful step is to let Rust pass item metadata, not just labels.

Minimum useful fields:

- `key`
- `label`
- `selected`
- `depth` for world tree indentation
- `action_kind`

If MMS object literals stay awkward, this can still be encoded with parallel arrays or by generating row names directly in the script source Rust emits.

## Non-Goals For The First Pass

- Do not make scroll position or pointer hover drive rerenders.
- Do not expose the full editor API directly to MMS handlers yet.
- Do not require the factory to know about `ComponentId` values.

## Next Implementation Step

After the current subtree-swap refactor settles, add a host-side row binding phase:

1. render panel asset
2. query row nodes by stable item key
3. register Rust click handlers against those row nodes

That gives us full MMS-authored structure with host-owned editor behavior, which is the right split for the panel system in its current state.
