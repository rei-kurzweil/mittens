# Meow Meow Script — Objectives

This document describes **what MMS is trying to be and why**, separate from how it is implemented.

---

## What is MMS?

Meow Meow Script is a **purpose-built authoring language for cat-engine**.

It is not a general-purpose scripting language bolted onto a game engine. It is not a configuration format. It is not a template language. It is a small language whose central construct — the *component expression* — maps directly and cleanly onto cat-engine's ECS component-tree model.

---

## Design objectives

### 1. Make scene authoring readable and writable by Rei

The primary author is Rei. The language should feel natural to write by hand in a text editor, not generated by a tool. Every syntax decision should pass the "would Rei write this comfortably?" test.

This means:
- Minimal ceremony. No mandatory type annotations, no boilerplate wrappers.
- Whitespace-insensitive. Indentation is for humans, not the parser.
- Short forms for common components (`T`, `R`, `C`, `BGC`, ...) without sacrificing readability of the tree structure.

### 2. Make component trees look like trees

A component expression should read like the structure it describes. Nesting = parent/child. The visual indentation of the source file should match the conceptual hierarchy of the scene.

```
CTLXR.new(true, Left, Aim) {
    T.with_scale(0.06, 0.06, 0.12) {
        TransformPipeline {
            TransformForkTRS {
                TransformMapTranslation {}
                TransformMapRotation {
                    QuatTemporalFilter.with_smoothing_factor(220.0)
                }
            }
        }
    }
}
```

This should be the *natural* way to write a scene — not a special mode or a macro, just the language.

### 3. Constructor calls, not data literals

Component expressions look declarative but are not data. Every node is a constructor invocation. `Renderable.cube()` and `Renderable {}` are different things — one calls a named factory, one calls the default constructor.

The language owns this distinction rather than hiding it. Named constructors, builder patterns, and factory methods are first-class syntax (the "constructor call": `Type.method(args)`), not awkward edge cases.

### 4. Be the scene format (replace JSON)

MMS `.mms` files replace the JSON scene serialization. The source of truth for what a scene contains is the `.mms` file, not a generated JSON snapshot.

This means:
- Round-trip fidelity: a live component tree can be encoded back to MMS (un-parsing / pretty-printing).
- The MMS file is the thing you commit to git, not the serialized state.
- The REPL can read MMS snippets live, not just JSON.

### 5. Be a transpilation source

MMS is designed to eventually be a source language that can emit to multiple targets:
- cat-engine ECS (the runtime — component trees, intents)
- Rust source (for when you want generated Rust rather than a runtime interpreter)
- Potentially: other component systems or scene formats

This shapes some decisions: the AST is kept clean and re-emittable. The language avoids runtime-only concepts that can't be expressed statically. The un-parser (`MmsPrinter`) is a first-class citizen.

### 6. Stay small

MMS is not trying to be a full programming language. The imperative layer (`let`, `if`, `return`, function definitions) is a convenience layer for expressing dynamic scenes — not a goal in itself.

The core value is the component expression. Everything else is scaffolding that lets you parameterize and compose component expressions.

**MMS should not grow features for their own sake.** If something can be handled in Rust, it should be. MMS is for things that benefit from the declarative tree syntax and the authoring-time readability.

### 7. Host-defined, not self-sufficient

MMS does not define what `Transform`, `Renderable`, or `Color` are. The host (cat-engine) provides a component registry that maps type names to constructors. The language provides the calling convention; the host provides the vocabulary.

This is intentional: it keeps the language small and means the same MMS runtime could, in principle, drive a different host that registers different components.

---

## What MMS is not

- **Not a general scripting language.** Lua, Wren, and Rhai exist. MMS is not competing with them and is not trying to be embeddable in other projects.
- **Not a markup language.** It is not HTML, XML, or a configuration format. There are no attributes, no schemas, no validators separate from the evaluator. It is code.
- **Not a data serialization format.** JSON, TOML, RON exist. MMS files can *represent* scene data, but MMS is a language that *constructs* things, not a format that *describes* things.
- **Not complete.** v1 is intentionally minimal. The goal is to get component expressions working end-to-end. The language will grow, but only as needed.

## Relationship to the engine

```
.mms source
    ↓ tokenizer + parser
AST (ComponentExpression, Statement, ...)
    ↓ evaluator (stop-gap: BuildCommand; final: SpawnComponentTree intent)
Engine component tree (live ECS nodes, parent/child topology)
    ↓ un-parser (MmsPrinter)
.mms source  ← round-trip
```

The evaluator runs on the MMS worker thread and communicates with the engine main thread via intents (`SpawnComponentTree`). It never directly touches the ECS; it only emits intents.

The un-parser runs on the main thread (it reads live component state) and produces a new AST or source string. It is the serialization path.
