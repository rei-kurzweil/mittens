# Assets selection and paint panels

## Context

We want an editor workflow for asset discovery, preview, selection, and surface painting in the engine.
The first phase is the assets panel, which scans `assets/components/` for `.mms` files and exposes exported factory functions as assets.
This should be the baseline before adding the paint tool, selection models, and grid placement helpers.

This is a strong opportunity to leverage a major advantage of the engine: MMS files can contain both the "scene" and the factory logic in one human-readable module, instead of requiring separate script and scene files.

## Phase 1: Assets panel + previewable exported factory assets

### Goal

Build an Assets Panel that enumerates assets from `assets/components/` and renders a preview for each factory export.

### Requirements

- Read `assets/components/` for `.mms` files.
- Identify exported functions in each module (named exports).
- For each exported function, capture:
  - the function name
  - the parent `.mms` file
- Instantiate one preview of the exported component/factory and render it in the panel.
- Display each asset in a square preview tile with the component name below it.
- Use a grid / inline-block layout so the assets panel looks like a Unity-style asset browser.

### Notes

- The initial implementation can hardcode `assets/components/` as the scan path.
- Later we can make this path configurable, but not for phase 1.
- The asset discovery and enumeration should live in an assets system, not be baked into a stopgap MMS adapter.
- Each asset is a factory/prefab: a callable MMS export that returns a component tree.
- The preview should be generated by actually invoking the factory and rendering the result in a small preview viewport.

## Phase 2: Selection model and panel interaction

### Goal

Create selection models for the Assets Panel and the Paint Panel, and keep painting distinct from normal scene selection.

### Requirements

- Add a UI Selection model for list-based panels.
- The Selection component should:
  - wrap selectable items
  - emit `SelectionChanged` events within its scope
  - optionally move a visible selection indicator over the selected item
  - compute the selected item's bounding box/dimensions for highlight and focus behavior
- Assets Panel selection is separate from Paint Panel selection.
- Painting should be a distinct mode/tool, not the default click behavior for the scene.
- Panel selection should be able to select:
  - the active asset in the Assets Panel
  - the active tool in the Paint Panel

### Notes

- The selection model should support selecting a panel item and also selecting a tool inside the Paint Panel.
- The next phase should only activate painting when the paint tool is selected.

## Phase 3: Paint panel surface placement

### Goal

Add a paint tool that places objects onto surfaces, aligned to surface normals and respecting bounding boxes.

### Requirements

- The Paint Panel should allow painting the currently selected asset onto a target surface.
- Placement should examine the surface normal and orient the painted object to match the inclination of the surface.
- The system should check bounding boxes of both source and target objects to avoid overlaps at placement.
- Add a toggle for placement mode:
  - attach-painted-object: make the painted object a child of the surface object and inherit its world transform
  - align-only: place the object aligned to the surface but leave it detached from the surface's transform tree
- The paint tool should work before grid snapping is introduced.

### Notes

- This is the core of the paint workflow: choose an asset, choose paint mode, click a surface, create a placed object.
- The Paint Panel is also a panel with its own selection model for tool choices.
- The paint tool should be able to place objects without switching the normal scene selection behavior when painting is not active.

## Phase 4: Grid helpers and procedural placement

### Goal

Add flexible grid helpers for editor placement, snapping, and procedural placement.

### Requirements

- Support zero or more grids in the scene.
- Grids may overlap, have different angles, and different granularity.
- Grids apply only during editor placement / build time, unless explicitly used at runtime.
- Grids should be selectable independently of normal scene objects.
- Each grid should have:
  - a transform
  - a size/granularity parameter (e.g. world unit cell size)
  - a snap mode that affects painted objects when that grid is selected
- Grids should not force painted objects to become nested inside the grid transform in the live scene.
  - The grid may be owned by a transform, but painted children should ignore that transform for runtime inheritance unless intentionally parented.

### Notes

- Example: a 1m × 1m placement grid and a 2m × 2m placement grid can both exist, with different orientation and snapping.
- This is an editor-only helper layer; once objects are placed, the grid is not part of their behavior unless explicitly retained.

## Why this order?

1. The assets panel is the obvious first step: without an asset browser and preview, there is nothing to paint.
2. Selection models are needed so the system knows which asset and which paint tool are active.
3. Surface placement is the first real painting behavior and requires anchor/normal alignment and attach-vs-align semantics.
4. Grid helpers are powerful, but should come after a working paint workflow so they can be used as a placement enhancement rather than the core mechanic.
